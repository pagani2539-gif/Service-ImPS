# รายงานผลการตรวจสอบและปรับปรุงระบบ (System Optimization & Bug Report - May 2026)

**วันที่จัดทำ:** 23 พฤษภาคม 2026  
**สถานะ:** ดำเนินการเสร็จสิ้น (Implemented & Verified)

## 1. การแก้ไขปัญหา "ภาพหายช่วงเที่ยงคืน" (Midnight Image Loss Fix)
*   **สาเหตุของปัญหา:** เกิดจาก Resource Contention (การแย่งทรัพยากร) ระหว่างฟังก์ชันลบรูปเก่า (Cleanup) และการบันทึกรูปใหม่ที่รันพร้อมกันตอน 00:00 น. ทำให้ Disk/DB ทำงานหนักจน Snapshot Matching เกินเวลา Timeout
*   **การแก้ไขที่ดำเนินการ:**
    *   **Reschedule Cleanup:** เลื่อนเวลาการลบรูปภาพและข้อมูลเก่าจาก 00:00 น. ไปเป็น **04:00 น.** (ช่วงรถน้อยที่สุด)
    *   **Increased Timeout:** ปรับค่า `SNAP_MATCH_MAX_WAIT_MS` จาก **3s เป็น 5s** เพื่อให้ระบบมีเวลารอรูปภาพในช่วง Peak Load
    *   **Database Indexing:** เพิ่ม Index ให้คอลัมน์ `stamp` ในตาราง `snapshots` ทำให้การ Query และการ Delete ข้อมูลเร็วขึ้นอย่างมหาศาล (ทำโดยผู้ใช้งาน)

## 2. การเพิ่มประสิทธิภาพการประมวลผล (Performance Optimization)
*   **ปัญหาเดิม:** ระบบค้นหาและ Upload รูปแบบ Sequential (ทีละขั้นตอน) หากรูป Overview ช้า จะทำให้การส่ง OCR รูป LPR ช้าตามไปด้วย (คอขวด)
*   **การแก้ไขที่ดำเนินการ (Parallel Snapshotting):**
    *   แยกการทำงานของ **LPR และ Overview** ออกเป็นอิสระต่อกัน (Independent Promises)
    *   **LPR Priority:** เมื่อพบรูป LPR ระบบจะส่งไป OCR และ Upload ทันทีโดยไม่ต้องรอรูป Overview
    *   **ผลลัพธ์:** ป้าย LED แสดงผลเร็วขึ้น และลดความหน่วงใน Main Loop ของ Controller

## 3. รายงานการตรวจสอบ Bug (Bug Fix Report)
จากการตรวจสอบโค้ดเชิงลึก พบและแก้ไขจุดบกพร่องดังนี้:

| จุดที่พบ (Location) | อาการของ Bug (Issue) | ผลการแก้ไข (Fix Result) |
| :--- | :--- | :--- |
| **Retry Logic** | ระบบจะ Retry หารูปใหม่เฉพาะกรณีที่ "หายทั้งคู่" เท่านั้น ถ้าใบใดใบหนึ่งหายระบบจะไม่ตามหา | แก้ไขให้ Retry ทันทีหาก **หายใบใดใบหนึ่ง** (`||`) ทำให้ข้อมูลรูปภาพสมบูรณ์ขึ้น |
| **DataLogger Context** | พบการปิดวงเล็บและ Catch Block ซ้ำซ้อนจากการ Refactor ครั้งก่อน | ทำการคลีนโค้ดและตรวจสอบ Syntax ให้ถูกต้องตามมาตรฐาน Node.js แล้ว |
| **I/O Contention** | การสแกน Disk เพื่อลบรูปจำนวนมากในวินาทีแรกของวัน | เลื่อนเวลาเป็น 04:00 น. และใช้ระบบ Non-blocking I/O เต็มรูปแบบ |

---

## 4. สถานะความพร้อมของระบบ (System Health)
*   **Syntax Check:** ผ่านการตรวจสอบ `node --check` ทุกไฟล์ (DataLogger, InterComp, CleanupService, SnapshotManager)
*   **Database:** มี Index รองรับการค้นหาและลบข้อมูล (idx_snapshots_stamp)
*   **Concurrency:** รองรับการประมวลผลรถหลายคันพร้อมกันได้เสถียรขึ้น

---
**ข้อเสนอแนะเพิ่มเติม:**  
ควรเฝ้าระวังพื้นที่ว่างใน Disk (Disk Space) หากจำนวนรถต่อวันเพิ่มสูงขึ้นมาก เนื่องจากมีการเพิ่ม Timeout และการ Retry ที่เข้มงวดขึ้นอาจทำให้มีการเก็บรูปภาพได้ครบถ้วนกว่าเดิม ซึ่งส่งผลต่อการใช้พื้นที่ Disk ครับ
